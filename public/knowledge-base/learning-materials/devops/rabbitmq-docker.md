# RabbitMQ Setup using Docker

A comprehensive, production-grade guide to deploying, configuring, and managing RabbitMQ instances and clusters using Docker and Docker Compose.

---

## Introduction

RabbitMQ is one of the most popular open-source message brokers. It supports multiple messaging protocols and is highly lightweight and easy to deploy on-premises and in the cloud. 

Deploying RabbitMQ with Docker simplifies setup, ensures consistency across environments, and isolates configuration. This guide covers single-container setups, customized configurations, and high-availability clustered deployments.

---

## 1. Quick Start (Single Instance)

To run a single RabbitMQ instance with the Management Web UI enabled, use the official `rabbitmq:management` image.

```bash
docker run -d \
  --name rabbitmq-server \
  -p 5672:5672 \
  -p 15672:15672 \
  -e RABBITMQ_DEFAULT_USER=admin \
  -e RABBITMQ_DEFAULT_PASS=SuperSecurePass123 \
  rabbitmq:3.13-management
```

### Key Ports Explained
*   `5672`: Standard AMQP client port.
*   `15672`: Management UI and HTTP API port (username: `admin`, password: `SuperSecurePass123`).
*   `1883` (Optional): MQTT client port (if MQTT plugin is enabled).
*   `15675` (Optional): Web MQTT client port.

## 2. Production Docker Compose Setup

For production or persistent development setups, use **Docker Compose**. This ensures that data, logs, and custom configurations survive container restarts.

### Directory Structure
```text
rabbitmq-setup/
├── docker-compose.yml
├── config/
│   ├── rabbitmq.conf
│   └── enabled_plugins
└── data/ (Auto-generated by Docker for persistence)
```

### `docker-compose.yml`
Create a `docker-compose.yml` file to define the service, volume mounts, and resource constraints:

```yaml
version: '3.8'

services:
  rabbitmq:
    image: rabbitmq:3.13-management-alpine
    container_name: rabbitmq-prod
    ports:
      - "5672:5672"
      - "15672:15672"
      - "15692:15692" # Prometheus metrics
    environment:
      RABBITMQ_DEFAULT_USER: admin
      RABBITMQ_DEFAULT_PASS: ProdSecurePassword99!
      RABBITMQ_DEFAULT_VHOST: /
    volumes:
      - ./data:/var/lib/rabbitmq
      - ./config/rabbitmq.conf:/etc/rabbitmq/rabbitmq.conf:ro
      - ./config/enabled_plugins:/etc/rabbitmq/enabled_plugins:ro
    restart: always
    deploy:
      resources:
        limits:
          cpus: '2.0'
          memory: 2G
        reservations:
          memory: 512M
```

## 3. Configuration & Tuning

RabbitMQ uses configuration files to define limits and server behavior. Under Docker, we can mount these files dynamically.

### `rabbitmq.conf` (Configuration File)
Create a file under `config/rabbitmq.conf` to configure memory limits, disk limits, and performance options:

```ini
# Core Configuration
listeners.tcp.default = 5672

# Memory & Flow Control
# Trigger flow control if memory usage exceeds 40% of available system memory
vm_memory_high_watermark.relative = 0.4

# Disk Free Space Limit
# RabbitMQ will block producers if free space drops below 2GB
disk_free_limit.absolute = 2GB

# Networking / Keepalives
heartbeat = 60
```

### `enabled_plugins` (Plugins File)
Create a file under `config/enabled_plugins` to pre-activate plugins without running container shell commands:

```erlang
[rabbitmq_management, rabbitmq_prometheus].
```

> [!TIP]
> The `rabbitmq_prometheus` plugin exposes Prometheus metrics on port `15692` under `/metrics`. You can plug this directly into a Grafana dashboard.

## 4. Advanced: Multi-Node Clustering

To achieve high availability and failover capability, you can run a clustered RabbitMQ setup. In a cluster, all nodes share schema details, but queues can mirror messages across nodes.

### Cluster `docker-compose.yml`

```yaml
version: '3.8'

services:
  rabbitmq-node1:
    image: rabbitmq:3.13-management-alpine
    container_name: rabbitmq-node1
    hostname: rabbitmq-node1
    ports:
      - "5672:5672"
      - "15672:15672"
    environment:
      RABBITMQ_ERLANG_COOKIE: "RabbitMQSuperSecretCookieShared"
      RABBITMQ_DEFAULT_USER: admin
      RABBITMQ_DEFAULT_PASS: cluster-secret
    volumes:
      - node1_data:/var/lib/rabbitmq
    networks:
      - rabbitmq-net

  rabbitmq-node2:
    image: rabbitmq:3.13-management-alpine
    container_name: rabbitmq-node2
    hostname: rabbitmq-node2
    environment:
      RABBITMQ_ERLANG_COOKIE: "RabbitMQSuperSecretCookieShared"
    volumes:
      - node2_data:/var/lib/rabbitmq
    networks:
      - rabbitmq-net
    depends_on:
      - rabbitmq-node1

  rabbitmq-node3:
    image: rabbitmq:3.13-management-alpine
    container_name: rabbitmq-node3
    hostname: rabbitmq-node3
    environment:
      RABBITMQ_ERLANG_COOKIE: "RabbitMQSuperSecretCookieShared"
    volumes:
      - node3_data:/var/lib/rabbitmq
    networks:
      - rabbitmq-net
    depends_on:
      - rabbitmq-node2

volumes:
  node1_data:
  node2_data:
  node3_data:

networks:
  rabbitmq-net:
    driver: bridge
```

### Joining Nodes to the Cluster

After spinning up the containers using `docker compose up -d`, you need to configure the secondary nodes to join `rabbitmq-node1`.

#### Configure Node 2:
```bash
docker exec -it rabbitmq-node2 rabbitmqctl stop_app
docker exec -it rabbitmq-node2 rabbitmqctl join_cluster rabbit@rabbitmq-node1
docker exec -it rabbitmq-node2 rabbitmqctl start_app
```

#### Configure Node 3:
```bash
docker exec -it rabbitmq-node3 rabbitmqctl stop_app
docker exec -it rabbitmq-node3 rabbitmqctl join_cluster rabbit@rabbitmq-node1
docker exec -it rabbitmq-node3 rabbitmqctl start_app
```

Verify cluster status on any node:
```bash
docker exec -it rabbitmq-node1 rabbitmqctl cluster_status
```

## 5. Troubleshooting & Best Practices

> [!WARNING]
> **Erlang Cookie Permissions**: When setting up manual volumes for Erlang cookies or configurations, ensure directory ownership aligns with the RabbitMQ UID/GID (`999:999`).

### Common Issues

| Issue | Root Cause | Solution |
| :--- | :--- | :--- |
| **Authentication Fails** | Wrong credentials or cache issue. | Clear client caches, double-check env variables or reset via `rabbitmqctl change_password`. |
| **Container Loops/Crashes** | Insufficient disk space or memory alarm triggered. | Free disk space or increase `disk_free_limit` / `vm_memory_high_watermark`. |
| **Connection Timeout** | Port `5672` is blocked or mapped incorrectly. | Check mapping in `docker-compose.yml` and run `telnet localhost 5672`. |

### Performance Checklist
1. **Never use default guest/guest** credentials in production.
2. **Monitor File Descriptors**: Ensure the host has a high enough limit for network connections (at least 65,536).
3. **Use quorum queues** instead of classic mirrored queues for modern, reliable clustering.
